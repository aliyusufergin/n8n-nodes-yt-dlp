# Version platform packages in lockstep

The main node package and both architecture-specific platform packages will carry the same SemVer version. The main package will reference the platform packages as optional dependencies using exact versions, never ranges. Release automation will publish and verify both platform packages before publishing the main package, and failure on either architecture will prevent that main release. Published artifacts are immutable; corrections require a new coordinated version. Initial packages will use `0.1.0` and the `next` dist-tag until real registry installation and both architecture paths pass, after which `1.0.0` becomes `latest`. Thereafter patches preserve the node and output contract, minors add backward-compatible interface or catalog capability, and majors cover ordinary breaking changes or a narrowed runtime. An urgent vulnerability may exceptionally restrict behavior in a patch with an advisory and deprecation of affected releases. Tool component versions remain explicit in the manifest and execution result rather than being inferred from the npm package version.

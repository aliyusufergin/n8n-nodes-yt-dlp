# v0.2.0 release definition

Accessed: 2026-07-17

## Basis

This note consolidates accepted ADR 0001 through ADR 0031. It does not add implementation evidence or claim that any release gate has already passed.

## Findings

1. **Ürün kararı:** `0.2.0` is release-ready only as one exact three-package Release Candidate Chain installed from the public npm registry under `next`; local workspaces, packed-but-unpublished tarballs, PATH tools, and manually altered containers are not acceptance evidence. The `latest` tag remains blocked until every gate below passes against the same immutable bytes. Basis: [ADR 0010](../adr/0010-three-package-platform-selection.md), [ADR 0023](../adr/0023-staged-lockstep-npm-release.md), and [ADR 0028](../adr/0028-linux-x64-platform-and-install-gate.md).

2. **Ürün kararı:** The disposable release matrix must pass Community Packages install/loading and the full hermetic queue-mode node contract on exact official Linux x64 image digests for n8n 2.0.0, the acceptance anchor 2.27.4, and the frozen newest stable 2.x anchor. The head lane additionally proves two-worker propagation, empty-package worker recreation, and late-worker recovery. The separate live, no-download JSC canary must pass cleanly; `inconclusive` does not permit promotion. Basis: [ADR 0025](../adr/0025-three-anchor-n8n-2x-release-gate.md) and [ADR 0030](../adr/0030-hermetic-e2e-and-live-jsc-canary.md).

3. **Ürün kararı:** Because the existing Docker queue-mode stack is the first-class acceptance environment, promotion also requires one separately approved, operation-specific E2E against its exact n8n 2.27.4 image and configured database binary backend. Using the exact `next` packages, it proves GUI/backend install, package propagation to each existing worker, one production and one worker-offloaded manual execution, packaged yt-dlp/FFmpeg/FFprobe/Deno selection, binary round-trip, bounded cancellation, secret-safe logs, and owned-workspace cleanup. It does not recreate containers, add workers, change configuration, run a capacity/failure load, or use third-party credentials/content; those disruptive cases remain in disposable CI unless separately approved. No acceptance-server operation is authorized by this decision itself.

4. **Ürün kararı:** Security and process acceptance requires exhaustive/fuzzed Arguments Grammar and allowlist tests, fixed output confinement and symlink/path defenses, bounded process output, whole Process Group cancellation/timeout, Resource Envelope enforcement, secret transport/redaction, Toolchain Attestation, deterministic cleanup, and stable Result/log schemas. Each forbidden capability must have a negative test proving no child process starts when rejection is expected. Basis: [ADR 0014](../adr/0014-n8n-credential-and-stdin-secret-config.md) through [ADR 0022](../adr/0022-worker-local-workspace-recovery.md), [ADR 0026](../adr/0026-stable-workflow-result-contract.md), [ADR 0029](../adr/0029-lazy-runtime-toolchain-attestation.md), and [ADR 0031](../adr/0031-bounded-logs-and-operator-observability.md).

5. **Ürün kararı:** Supply-chain and license acceptance requires a frozen Toolchain Lock, verified upstream checksums/signatures where available, reproducible package inventory, exact tarball digests, provenance, no lifecycle-script dependency, component license/notice surface, a clean rebuild from the exact Corresponding Source Bundle, and manual license review before the GPL platform package is published. Any missing source or unexplained binary blocks the whole chain. Basis: [ADR 0012](../adr/0012-pinned-gpl-ffmpeg-build.md), [ADR 0013](../adr/0013-immutable-nightly-toolchain.md), [ADR 0024](../adr/0024-component-license-surface-and-source-gate.md), and [ADR 0029](../adr/0029-lazy-runtime-toolchain-attestation.md).

6. **Ürün kararı:** Documentation acceptance requires an exact supported/verified matrix, Community Packages install instructions, node/credential/Result contracts, hard caps and cancellation behavior, Network Trust Boundary, database binary-storage requirement, package and source digests, license/source locations, operator monitoring/capacity/cleanup duties, and all remaining unverified assumptions. Examples use only synthetic or clearly authorized content and never imply a supported-site guarantee.

7. **Ürün kararı:** The v0.2.0 product scope is one programmatic regular node on official n8n Docker Linux x64, n8n `>=2 <3`, queue mode with database binary storage, sequential per-input HTTP(S) Download Requests, the exact V1 Argument Allowlist, optional n8n Authentication Credential, packaged yt-dlp/FFmpeg/FFprobe/Deno, atomic per-request Artifact Items, Continue On Fail request errors, bounded resources/process output, cancellation, and worker-local cleanup.

8. **Ürün kararı:** Explicitly out of scope are n8n 1.x/3.x and Cloud; arm64, Windows, macOS, bare metal, non-official images, and other libc/container bases; S3/external binary storage; AI Agent tool use; parallel request execution; partial-success Artifact publication; arbitrary CLI/shell/config/plugin/self-update/external executable access; PATH/system-tool or runtime-download fallback; arbitrary headers/tokens, browser-cookie import, OAuth/OTP/client certificates; application-level SSRF protection for untrusted URLs; transactional binary rollback; same-UID/root sandboxing; immediate cleanup after SIGKILL/host failure; node-owned metrics/telemetry; universal capacity thresholds; and any supported-site or uninterrupted live-extractor guarantee.

9. **E2E ile doğrulanacak varsayım:** None of the accepted design is release evidence yet. Published-package install semantics, exact executable compatibility, generic synthetic extractor fixtures, JSC evidence, MIME mapping, log cardinality, concurrency-ten capacity, database transfer memory behavior, and actual acceptance-stack operation remain **doğrulanmadı** until their specified gates run.

10. **Lisans/güvenlik riski:** A green unit suite, a local `npm pack`, a single direct-file download, or a successful live-site extraction can each conceal a release-blocking failure in a different boundary. No individual lane may waive another, and a failed gate must change code/package/topology or the documented scope rather than be relabeled as accepted.

11. **Cevapsız soru:** Exact fixture formats, final Toolchain Lock identities, source-bundle inventory, concrete capacity/alert thresholds, live canary identity/retry window, and operation-specific acceptance-server commands are **doğrulanmadı**. They must be resolved in implementation/release issues; every acceptance-server state change still requires the prescribed seven-part plan and fresh explicit approval.

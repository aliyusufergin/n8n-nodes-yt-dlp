# n8n 2.27.4 Community Packages and queue mode

Accessed: 2026-07-17

Scope: the installation, persistence, and worker-loading behavior that constrains an npm-installed `n8n-nodes-yt-dlp` package, plus queue-mode binary-data and cancellation behavior. Product decisions are deliberately excluded.

## Version anchors

- n8n tag: `n8n@2.27.4`, commit [`a4d0dfce294064026be1a6a246e6da348fea1485`](https://github.com/n8n-io/n8n/tree/a4d0dfce294064026be1a6a246e6da348fea1485).
- n8n documentation snapshot: commit [`c130d433dff64e1ece2e84cfa4658ccad127794e`](https://github.com/n8n-io/n8n-docs/tree/c130d433dff64e1ece2e84cfa4658ccad127794e). The public documentation is rolling rather than versioned to n8n 2.27.4, so source/doc differences below are risks rather than a combined contract.

## Findings

### Package location and cluster propagation

1. **Kanıtlanmış platform gerçeği:** Each n8n process resolves its Community Packages directory as `<n8nFolder>/nodes`; loaders scan `<n8nFolder>/nodes/node_modules`. Package files are therefore instance-filesystem state, while installed-package and installed-node metadata are database entities. Sources: [`InstanceSettings.nodesDownloadDir`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/core/src/instance-settings/instance-settings.ts), [`CommunityPackagesModule.nodeLoaders`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.module.ts), [`InstalledPackages`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/installed-packages.entity.ts), and [`InstalledNodes`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/installed-nodes.entity.ts), all at n8n 2.27.4 commit `a4d0dfce`.

2. **Kanıtlanmış platform gerçeği:** After a successful install/update and database save, `CommunityPackagesService` publishes `community-package-install` or `community-package-update` with package name and requested version. A receiving process handles those events by downloading, unloading, and loading the package in its own package directory. Sources: [`CommunityPackagesService.installOrUpdatePackage` and event handlers](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.service.ts), [`PubSub event map`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/scaling/pubsub/pubsub.event-map.ts), [`Publisher`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/scaling/pubsub/publisher.service.ts), and [`Subscriber`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/scaling/pubsub/subscriber.service.ts), all at commit `a4d0dfce`.

3. **E2E ile doğrulanacak varsayım:** The source flow is intended to make every online worker install and hot-load the same package version, but source inspection alone does not prove delivery under worker restart, a worker joining late, a missed Pub/Sub event, or concurrent update. These cases require real Community Packages E2E against the supported queue topology.

4. **Kanıtlanmış platform gerçeği:** On startup, the service compares database metadata with known loaded node types. When `N8N_REINSTALL_MISSING_PACKAGES=true`, it attempts to install each missing package at its database-recorded exact version; the setting defaults to `false`. Sources: [`CommunityPackagesService.checkForMissingPackages`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.service.ts) and [`CommunityPackagesConfig`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.config.ts), n8n 2.27.4 commit `a4d0dfce`.

5. **E2E ile doğrulanacak varsayım:** `N8N_REINSTALL_MISSING_PACKAGES=true` is a recovery aid, not proof that package files persist or that a newly created worker becomes ready before accepting jobs. Readiness, exact-version reinstall, and failure behavior must be tested after recreating a worker with an empty package directory.

### npm installation semantics

6. **Kanıtlanmış platform gerçeği:** n8n 2.27.4 does not run a normal top-level `npm install <package>`. It runs `npm pack <name>@<version>`, extracts the tarball, removes the extracted package's `devDependencies`, `peerDependencies`, and direct `optionalDependencies`, then runs npm in that package directory with `--install-strategy=shallow`, `--ignore-scripts=true`, `--bin-links=false`, and no package lock. Source: [`CommunityPackagesService.downloadPackage`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.service.ts), n8n 2.27.4 commit `a4d0dfce`.

7. **Kanıtlanmış platform gerçeği:** A platform binary package placed directly in the main package's `optionalDependencies` cannot be selected by this installer because n8n removes that field before npm resolves dependencies.

8. **E2E ile doğrulanacak varsayım:** Keeping platform packages in a regular selector dependency and placing `os`/`cpu`-filtered platform packages in the selector's own `optionalDependencies` is consistent with the observed prototype, but neither n8n source nor npm metadata documentation guarantees the complete n8n-specific result. It must pass real GUI install on every claimed architecture, including absence of nonmatching packages and executable startup on the worker. npm metadata reference: [`package.json` fields, including `os`, `cpu`, and `optionalDependencies`](https://docs.npmjs.com/cli/v11/configuring-npm/package-json/).

9. **Lisans/güvenlik riski:** `--ignore-scripts=true` means packages cannot rely on lifecycle scripts to download, compile, chmod, or repair bundled executables. `--bin-links=false` means the design cannot rely on npm-generated command shims. Executables and permissions must already be correct in published tarballs and must be invoked through resolved absolute paths.

### Queue-mode binary data

10. **Kanıtlanmış platform gerçeği:** n8n 2.27.4 source defaults `N8N_DEFAULT_BINARY_DATA_MODE` to `database` when execution mode is `queue`, and to `filesystem` otherwise. Database mode has a configurable maximum of 512 MiB by default and a schema ceiling of 1024 MiB. Source: [`BinaryDataConfig`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/core/src/binary-data/binary-data.config.ts), n8n 2.27.4 commit `a4d0dfce`.

11. **Kanıtlanmış platform gerçeği:** Both official docs pages say filesystem binary-data mode is unsupported in queue mode. The binary-data page says to use `database`; the queue-mode page instead directs persistent binary data to S3. Sources: [`handle-binary-data.md`](https://github.com/n8n-io/n8n-docs/blob/c130d433dff64e1ece2e84cfa4658ccad127794e/docs/deploy/host-n8n/configure-n8n/scaling/handle-binary-data.md) and [`enable-queue-mode.md`](https://github.com/n8n-io/n8n-docs/blob/c130d433dff64e1ece2e84cfa4658ccad127794e/docs/deploy/host-n8n/configure-n8n/scaling/enable-queue-mode.md), docs commit `c130d433`.

12. **Lisans/güvenlik riski:** The two docs pages are inconsistent about the recommended queue-mode persistence backend, and the rolling binary-data page also says binary data is kept in memory “by default,” which does not match the 2.27.4 queue-mode default in source. Architecture must use the tagged source for 2.27.4 behavior and separately E2E the configured backend. S3 external storage also has plan/licensing constraints documented at [`use-external-storage.md`](https://github.com/n8n-io/n8n-docs/blob/c130d433dff64e1ece2e84cfa4658ccad127794e/docs/deploy/host-n8n/configure-n8n/scaling/use-external-storage.md).

13. **Ürün kararı:** The node should hand artifacts to n8n through its binary-data helper APIs, not expose worker-local temporary paths as results. The supported artifact limits must fit the selected backend; source facts alone do not choose database versus external object storage.

### Cancellation

14. **Kanıtlanmış platform gerçeği:** The n8n 2.27.4 execution context exposes `getExecutionCancelSignal()` and `onExecutionCancellation()`, backed by an `AbortSignal`. Sources: [`BaseExecuteContext`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/core/src/execution-engine/node-execution-context/base-execute-context.ts) and [`WorkflowExecute`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/core/src/execution-engine/workflow-execute.ts), n8n 2.27.4 commit `a4d0dfce`.

15. **E2E ile doğrulanacak varsayım:** Forwarding this signal to a Node.js child process is necessary but not sufficient to prove FFmpeg descendants terminate. Timeout/cancel tests must verify the process tree, exit classification, temp cleanup, and absence of zombies in the actual Alpine worker container.

### Host n8n version compatibility

16. **Kanıtlanmış platform gerçeği:** The n8n 2.27.4 package loader reads the package name, package version, and `n8n.nodes`/`n8n.credentials` paths, but does not enforce a host-n8n semver range from community-package metadata. Source: [`PackageDirectoryLoader`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/core/src/nodes-loader/package-directory-loader.ts), n8n 2.27.4 commit `a4d0dfce`.

17. **Kanıtlanmış platform gerçeği:** The official starter package declares `n8n.n8nNodesApiVersion: 1` and `n8n-workflow: "*"` as a peer dependency, but no host-n8n version range. n8n 2.27.4 removes the main package's peer dependencies before installation, so that peer field cannot enforce host compatibility through the Community Packages installer. Sources: [`n8n-nodes-starter` package.json at commit `3308a8ec`](https://github.com/n8n-io/n8n-nodes-starter/blob/3308a8eca314e388c40b29c9b6cefc49a8cf9115/package.json) and [`CommunityPackagesService.downloadPackage`](https://github.com/n8n-io/n8n/blob/a4d0dfce294064026be1a6a246e6da348fea1485/packages/cli/src/modules/community-packages/community-packages.service.ts).

18. **Cevapsız soru:** Host compatibility therefore has to be a documented and tested support contract unless a separate, public n8n compatibility mechanism is found. A package manifest alone is not an enforcement boundary. As accessed on 2026-07-17, the latest official release is [`n8n@2.30.7`](https://github.com/n8n-io/n8n/releases/tag/n8n%402.30.7), while the acceptance server runs 2.27.4.

## Acceptance evidence still required

- Install the published main package through the real Community Packages GUI/backend on n8n 2.27.4; prove selector resolution and the chosen platform package on every supported architecture.
- Prove main and every existing worker load the exact version, then recreate/add a worker with no local package files and prove readiness/recovery behavior.
- Run a production and a manual execution (`OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true`) and prove the packaged yt-dlp/FFmpeg/FFprobe executables are selected on the worker.
- Store and retrieve artifacts at every declared count/size boundary through the configured queue-compatible binary backend; inspect neither DB credentials nor binary contents beyond test fixtures.
- Cancel and time out yt-dlp jobs that have and have not spawned FFmpeg; prove bounded logs, complete process-tree termination, and workspace cleanup.
